# Soak: the hostd memory leak hunt (2026-06-13)

The first compressed soak (sustained fsx + guest churn + Mac metadata storms, 25 min) failed the
"zero leak" gate: hostd RSS climbed **13 MB → 4.2 GB**. This is how it was root-caused and
fixed — recorded because the method (empirical localization + adversarial static audit) and
the false trails are worth keeping.

## Symptom

hostd RSS grew ~linearly, ~180 MB/min, under sustained single-file fsx on a v3 writeback
mount. A focused repro (tight fsx loop, one file) reproduced it at ~5 MB/s.

## Localization (empirical, not by reading)

1. **CAS not the (whole) story.** Repro logged CAS stats: blobs grew to the 20 GiB cap then
   evicted (bounded). But RSS kept climbing *past* the blob cap → a time-linear leak
   independent of blob count. redb file stayed **1.4 MB** (manifest superseding works), ruling
   out manifest/redb growth.
2. **`vmmap -summary`** at three time points: the growth was entirely **MALLOC_SMALL**
   (48–128 KB buffers, thousands of them), not thread stacks, not mapped files, not redb mmap.
   Thread count was ~28 (killed the "tokio blocking pool" theory).
3. **`malloc_history -allBySize`** named the exact site: **613 MB in 8063 live allocations**,
   all `…nfs3::dispatch_inner → ShareSurface::write → data.to_vec()`. The write payloads.
4. **Instrumented counters** showed the decisive fact: the backpressure semaphore stayed at
   96 free permits (writes weren't backing up in flight), but `writes` (the per-node
   write-behind map) grew **155 → 194 → … entries**, and each new entry had a **different
   inode with a random ext4 generation**.

## Root cause

fsx replaces its test file every run (write UNSTABLE, then unlink + recreate — exactly what
every editor's save-via-rename does). Each dead inode left its `NodeWrites` entry in the
`writes` map, and that entry holds the **entire stash of write payloads** for the file
(every WRITE, not deduplicated by offset — ~290 payloads/file). The only removal path was a
cleanup spawned on a *full COMMIT*, which never arrives for a file that's unlinked before
close. So one orphaned ~20 MB stash leaked per file, forever. 28 orphans × ~290 payloads
≈ the 8063 live buffers.

## Fixes (crates/mist-hostd/src/surface.rs)

1. **Idle reaper** (`spawn_write_reaper`): a 5 s background task drops `writes` entries that
   are idle (`inflight == 0`) and untouched for > 20 s. Bounds the map regardless of whether a
   COMMIT ever arrives. Holds a `Weak` so it exits with the surface.
2. **Immediate cleanup on unlink**: `remove()` drops the unlinked node's write-behind entry at
   once (single `writes` lock — see the deadlock note).
3. **Always-reclaim cleanup**: the COMMIT-time cleanup now spawns its entry-removal task even
   when the stash was abandoned over-cap (was gated behind `stash.is_some()`).
4. **Shared maps through `detached()`**: `writes`/`manifests` are now `Arc<Mutex<…>>` and
   cloned (not freshly-emptied) into the copies used by write-behind/cleanup tasks — the old
   code's cleanup removed from a throwaway empty map, so nothing was ever reclaimed.
5. **Write-behind backpressure** (`write_slots` semaphore, 96 permits): bounds in-flight
   UNSTABLE payloads at ≤96 MiB independent of the above. Not the dominant leak here (writes
   weren't backing up), but it's the correct guard against a client that outruns guest apply,
   and cheap insurance — knfsd throttles the same way via its bounded nfsd thread pool.

## The trap (recorded so it isn't repeated)

The first cut of fix #2 locked `self.writes` **twice in one `if let`**:
```rust
if let Some(q) = self.writes.lock().get(&n).cloned() && ... { self.writes.lock().remove(&n); }
```
`parking_lot::Mutex` is **not reentrant**, and the `if let` temporary guard lives to the end of
the statement → the second `.lock()` deadlocks the whole daemon. The repro then showed *flat*
RSS (67 MB) which looked like a perfect fix — but it was a hung process: blobs/hits frozen.
**Lesson: a leak repro must confirm the workload is still making progress** (fsx completions
climbing, cache evictions climbing), not just that RSS stopped growing. The corrected verify
(`leak-verify.sh`) asserts both.

## Result

Verified repro after the fix: RSS **flat at ~106 MB** over 4.5 min while fsx completions climb
24 → 179 and CAS evictions climb to 26k (healthy, active steady-state). Warm-read gate intact:
write-keep-remount-read serves from CAS with **0 guest reads**, sha-identical. 75 tests green,
clippy clean.

Tooling left in `e2e-work/`: `leak-repro.sh` (RSS+CAS over time), `leak-vmmap.sh`
(zone summary), `leak-history.sh` (MallocStackLogging backtraces), `leak-verify.sh`
(bounded-AND-active check).
