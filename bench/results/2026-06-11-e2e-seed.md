## Findings → bridge rewrite

- **Seed is the headline result and it clears the bar comfortably** (335k/s on a metadata-only
  walk in degraded-id mode; handle mode adds a `name_to_handle_at` per entry — re-measure on a
  privileged guest).
- **vsock RTT median is healthy (~125µs) but the tail is fat** (p99 4.9ms, max 33ms). The
  reference `MistBridge` is the suspect, not AVF vsock: it uses a thread-per-direction with
  256 KiB blocking `read`/`write` copies and a `DispatchSemaphore` per `connect`. The 5-ping warm
  probe showed p50 61µs, so the steady-state floor is lower; the tail is scheduling/jitter in the
  shim. **Next fix: replace the splice with a kqueue/`DispatchSource` pump and drop the per-connect
  semaphore; re-measure before drawing transport conclusions.**
- **Throughput (0.82 GiB/s, 4-lane) is well under the design's ≥1.5 GB/s aspiration** — same root
  cause (userspace-copy bridge). Not on the metadata critical path, but it gates large cold
  reads, so it rides with the bridge rewrite.
- **Decision recorded:** vsock is the default transport (works, sub-150µs median, no IP stack);
  TCP/virtio-net remains the fallback. AVF vsock itself was never the bottleneck in any failure —
  every transport bug was in our own glue (a Swift `&buf[off]` pointer footgun that corrupted
  the byte pipe; a missing endpoint-port parse; an over-strict echo-server classifier).
```

## Update — MistBridge DispatchSource rewrite (post-bridge-rewrite)

Replaced the thread-per-direction blocking splice (+ per-connect semaphore) with an event-driven
DispatchSource (kqueue) pump: non-blocking reads, a write source for backpressure, half-close
semantics, async connect (no semaphore). Re-measured on the same Apple Silicon Pro / Debian 13
guest:

| Metric | thread-per-dir | DispatchSource |
|---|---|---|
| RTT p50 | 125 µs | **62.6 µs** |
| RTT p90 | 612 µs | **69.2 µs** |
| RTT p99 | 4902 µs | **93.0 µs** (53× better) |
| RTT max | 33275 µs | **185.7 µs** (178× better) |
| throughput 1 lane | 0.15 GiB/s | **0.38 GiB/s** (2.5×) |
| throughput 4 lanes | 0.82 GiB/s | 0.86 GiB/s |

The fat RTT tail was GCD worker-thread wakeup jitter; the event-driven design removes it. The
4-lane plateau (~0.86 GiB/s) is unchanged → that ceiling is AVF vsock's aggregate limit, not the
bridge. Data path verified correct (seed, ls, cat, macOS NFS mount+read).
