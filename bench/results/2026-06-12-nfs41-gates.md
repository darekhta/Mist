# Mist NFSv4.1 + journal-driven delegations: gates (2026-06-12)

Host: macOS 26.5 (M-series), guest Debian 13 / 6.12.90 on AVF, loopback NFSv4.1
(`mist mount dev code --nfs41`). All numbers from the REAL macOS kernel client.

## R3 retired: macOS client validation (probe + real stack)

- `mount_nfs -o vers=4.1,tcp,port=N` mounts our server (macOS 26.5 supports minor version 1).
  `nolocks`/`locallocks` are v3-only and fail the mount with EINVAL — v4.1 mounts use neither.
- `nfsstat -m` shows `callback` — the client binds the v4.1 backchannel on the fore connection.
- Numeric owner strings ("501") map correctly; ls shows the right user/group.
- Client takes read delegations on read-opens and DELEGRETURNs on CB_RECALL.
- Client uses CLAIM_DELEG_CUR_FH to open under a held delegation, and expects proper
  open-owner stateid reuse (same stateid, bumped seqid, unioned access) — both implemented
  after wire traces showed recovery wedges.

## Client-quirk findings (wire-trace debugging)

1. OPEN_DOWNGRADE args are stateid-FIRST (unlike CLOSE) — parsing seqid first shifts the
   stateid 4 bytes and every downgrade is "unknown" → client enters reboot recovery
   (CLAIM_PREVIOUS) → NO_GRACE wedge.
2. Same open-owner re-opening a file must get the SAME stateid (bumped seqid, unioned
   access); minting per-OPEN stateids desyncs the client.
3. LOCK must succeed or the client fails the file with EIO; always-grant is sound for a
   single loopback client (v3 `locallocks` parity).
4. CLAIM_DELEG_CUR_FH / CLAIM_DELEGATE_CUR must be honored when delegations are on.
