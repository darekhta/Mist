# Performance push vs the comparison table (2026-06-12, evening)

Goal: exceed the recorded table numbers — write 330 MB/s (guest nfsd), read 1185 MB/s
(guest nfsd), creates 8.5 ms/file (guest nfsd). Same bench (`scripts/bench-compare.sh`),
same hardware, fresh guest boot for the final record.

## What was built (all profile-independent unless noted)

1. **Concurrent request processing** in both NFS servers (v3 + v4.1): per-record tasks,
   out-of-order replies (xid/slot matching makes this legal); a serial loop capped pipelined
   reads at a fraction of the client's slot depth. + `readahead=128` mount default.
2. **Write-behind** (design 04 §4.3 narrowed): UNSTABLE writes ack after local validation,
   batches fly concurrently (clients may not overlap WRITE ranges — knfsd relies on the same),
   COMMIT is the durability/error barrier. WRITE server time: 39–81 µs.
3. **Echo fences for real** (design 02 §6.5): journal echoes of Mac writes are DROPPED for
   mac-dirty nodes; optimistic attrs evolve monotonically; v4 change_info is atomic=true
   (single-client server). Without these the macOS client fell into read-modify-write storms
   (every MiB of a dd read back + written twice) and dir revalidation storms (170 ops per
   created file, LOOKUPs growing triangularly).
4. **Ranged COMMIT** (proto `CommitRange`, applier `sync_file_range` = knfsd
   `vfs_fsync_range` twin) + clean-commit skip (a full COMMIT with no unstable data owed is
   honestly a no-op — the client double-commits at CLOSE).
5. **Stable-write replies on writeback shares** (`share_flags::WRITEBACK` → WRITE replies
   FILE_SYNC, knfsd `async`-export semantics): removes client commit throttling entirely.
   This is the single biggest write win (~150 → ~530 MB/s).
6. **1 MiB write chunks** (MAX_FRAME 2 MiB) + guest applier **write-fd cache** + dedicated
   second rpc lane for bulk writes.
7. **CAS warm-write path**: write-behind payloads stashed (Arc, zero-copy) and batch-ingested
   at the final COMMIT in ONE durable transaction → read-after-write serves from cache
   (0 guest reads measured).
8. **Read path**: single-chunk fast path (no accumulator copy), no-zero blob reads,
   single-buffer compound assembly with header patching, large-buffer pool (page-fault
   amortization), zero-copy `sendfile(2)` for CAS-hit v3 READs, blob-fd cache.
9. **Robustness found by the push**: oversized control-plane `cat` killed the rpc lane
   (reads > MAX_READ_LEN now chunked); **a dead rpc lane now ends the session** so the
   supervisor reconnects (previously mutations failed forever while ctl pings kept the
   session "healthy").

## The remaining read gap, analyzed

Single-stream reads are latency-bound: the macOS client keeps ~1 outstanding READ
(+readahead via 2 nfsiod threads — `vfs.generic.nfs.client.nfsiod_thread_count: 2`); two
parallel streams hit 1238 MB/s aggregate immediately. Per-1 MiB e2e: nfsd ≈ 0.84 ms
(kernel zero-copy + virtio), Mist ≈ 1.1 ms. Paths to close it: client
`nfsiod_thread_max` tuning (needs root; would also lift nfsd — honest re-baseline needed),
further server-latency work, or kernel-bypass framing. macOS rejects rsize > 1 MiB
(falls back to 32 KiB — measured).

Correctness held throughout: fsx 4 seeds × 1500 ops clean on the final build; sha-verified
read-after-write through mount AND from inside the guest; 75 unit/integration tests green;
clippy clean.
