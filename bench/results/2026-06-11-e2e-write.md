## Verdict — write path: PROVEN on real hardware

A macOS app writing through the loopback NFS mount lands byte-for-byte in the guest's ext4,
read back over vsock with `mist cat` (which `open_by_handle_at`s + `pread`s the real inode):
create, write, append-edit, mkdir, nested write, rename, rm, and a 200 KB `cp` (sha **MATCH**).
Zero guest mutation failures. Squash works — the Mac sees the share owned by its own uid (501)
while the guest owns files as `apply_uid` (1000).

### Bugs found + fixed in this run
1. **`open_by_handle_at(O_WRONLY)` → EPERM on ext4.** Opening a file node *by handle* for write is
   rejected even with CAP_DAC_READ_SEARCH (only `O_PATH`/`O_RDONLY`-by-handle are allowed).
   Create/unlink/rename were unaffected because they use `*at` against the parent dir fd; only
   write/setattr opened the file node directly. Fix: open the privileged `O_PATH` handle (fsuid 0),
   then reopen `/proc/self/fd/N` as the apply identity for `O_WRONLY`/`O_RDONLY` — the standard
   O_PATH-upgrade pattern. Also fixed an EISDIR on directory `setattr` (reopen `O_RDONLY`, not
   `O_RDWR`). uid/gid in SETATTR are ignored under squash (the Mac sends its squashed owner).
2. **Identity squash** (the macOS client checks the *returned* attrs against the local Mac uid):
   `ShareSurface` rewrites every returned uid/gid to the Mac user (`geteuid`), and ACCESS grants
   MODIFY/EXTEND/DELETE on writable shares. Without it, `echo >` failed client-side with EACCES.

### e2e-harness lessons (real AVF, repeated boots)
- **Verify over vsock, not SSH.** macOS vmnet NAT reuses 192.168.64.3 across VM generations and
  leaves stale ARP, so SSH to the guest is flaky and was killing every run before it reached the
  write test. The whole write path is loopback+vsock (Mac→hostd→mistd); `mist cat`/`ls`/`stat`
  verify the guest's real ext4 with no guest networking. Dropped SSH entirely.
- **Stale NFS mounts hang the next run.** A killed hostd leaves a `127.0.0.1:/code` mount whose
  server is gone; with `hard,intr` a later `rm -rf "$STATE"` blocks forever. `clear_stale_mounts`
  (umount -f) now runs before touching state.
- **Readiness = the vsock seed, not a cloud-init console marker** (the marker is only printed on an
  instance's first boot; a reprovisioned disk skips runcmd but mistd still auto-starts).

### Observed release gaps at this point
- macOS writes AppleDouble `._<name>` sidecars (xattr/resource forks) into the share. Design 06's
  release side-store captures and garbage-collects AppleDouble metadata; at this point they
  appeared as real guest files.
